# MBank (Kyrgyzstan) — statement import

File-based community plugin. It imports an MBank (КР) account statement that the
user exports from the bank and selects manually. The file is parsed **entirely on
the device** — the plugin makes no network requests.

## How to export the statement from MBANK

1. Open the MBANK app.
2. Select the account you want a statement for.
3. Open **«История операций по счёту»** (account operation history).
4. Pick the period — **a single export covers at most 3 months**.
5. Tap the export icon (arrow, top-right corner).
6. Choose the **XLS** format and download the file.

Then pick the downloaded file in the plugin. To cover more than 3 months, export
several files and select them all at once — files of the same account are merged
and overlapping rows are deduplicated automatically.

## Supported file format

- `.xls` / `.xlsx` account statement ("Выписка из лицевого счета"), as produced by
  MBank (a binary XLS generated by JasperReports). One account / one currency per file.
- Multiple files can be selected at once — each becomes a separate account.

## How it works

1. `ZenMoney.pickDocuments` lets the user pick the statement file(s).
2. `parser-xls.ts` reads the sheet with `xlsx` and extracts, by label, the account
   number, currency and balance, then the transaction table
   (`Дата | Получатель/Плательщик | Дебет | Кредит | Операция`).
3. `converters.ts` maps each row to a ZenMoney `Transaction`:
   - debit → negative sum, credit → positive sum;
   - merchant/payee parsed from the description (card POS, QR/phone transfers,
     money envelopes, foreign transfers);
   - currency-conversion operations fill `movement.invoice` with the original amount.

### Transaction ids

The statement has no operation-reference column, so each movement id is a
**deterministic hash** (`cyrb53`) of the raw row
(`accountId | date-time | debit | credit | description`). The id is stable across
re-exports, which keeps deduplication working between syncs.

### Sync behaviour

Each statement file holds a single account. The plugin imports **everything in the
picked file** on every run:

- no first-run "accounts only" gate — transactions come on the very first run;
- no `fromDate` window — a manually chosen statement is imported in full, so an
  older statement never silently yields zero transactions. (The `startDate`
  preference is kept only because the platform requires it; it does not restrict
  what is imported.)

Re-importing the same (or an overlapping) file is safe: movements are deduplicated
by their deterministic id.

## Tests

```
yarn jest mbank-kg     # tests
yarn test mbank-kg     # typecheck + lint + tests
yarn start mbank-kg    # browser dev-harness
```

Tests run on an inline, anonymized fixture (`__tests__/parseStatementRows.test.ts`)
plus table-driven converter cases (`__tests__/converters.test.ts`) — no real
statement file (which contains personal data) is committed.
